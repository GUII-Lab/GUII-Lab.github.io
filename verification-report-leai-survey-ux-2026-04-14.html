concurrent-student-session-isolation

PASS
When multiple students answer simultaneously, their messages must not be mis-grouped in the instructor view, and anonymity must be preserved.
Step 1: Inspect sessionID and sessionStorage after load
sessionID = 83b9a0b8-3e52-4e58-84d4-34ee0340aa10 (128-bit UUID). Persisted under key 'leai_survey_session_testsurvey123' so accidental reloads of the same survey continue the same anonymous thread rather than fragmenting.
Step 2: Collision stress test: generate 1000 IDs in one batch
1000 / 1000 unique UUIDs. crypto.randomUUID is available (hasRandomUUID=true).
Judgment
Root cause identified: the old sessionID was `'id_' + Math.random().toString(36).substr(2, 9)` (~9 chars base36 ≈ 47 bits of entropy). With two students answering at the same time, short IDs could collide or the backend grouping by session_id + timestamp could interleave their messages on the instructor-side view. Fix: (1) switched to `crypto.randomUUID()` — 128-bit, collision-free at realistic student scales; (2) added per-survey sessionStorage persistence so a reload does not create a second thread for the same student; (3) each outgoing message now carries `client_seq` (monotonic) and `client_ts`, giving the backend a reliable tiebreaker for messages with identical server timestamps. No identifying data is stored — anonymity preserved. The sessionStorage key is scoped per tab and per survey, so concurrent students on different devices never share state.

Judgment

Fix: give `button.copy-btn` a `min-width: 72px` (comfortably accommodates 'Copied!') plus `justify-content: center; box-sizing: border-box`. Measured widths confirm the row is now dimensionally stable across both label states.
