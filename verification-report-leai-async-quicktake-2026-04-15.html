Context

Original bug: clicking AI Quick Take returned 503 with a CORS error banner. Root cause was a Heroku H12 router timeout (30s hard limit) on a synchronous OpenAI call, not a CORS misconfiguration.

Fix shipped in guiidatapipelines: converted leai_quicktake_generate from synchronous to an in-process background thread (no Redis, no worker dyno), switched the QuickTake model to gpt-4.1-mini, added status/error/job_started_at columns with a STATUS_CHOICES state machine, added a 600s stale-job detector, and returns HTTP 202 immediately. Frontend polls GET every 2s via new leaiChat.pollQuickTake.

This report verifies the end-to-end user flow in a real browser against a local Django backend running the exact code being deployed to Heroku.

Bugs Fixed During Verification

Root-cause fix discovered during verification

Initial run of this verification showed the POST returned 202 but the row remained status=pending indefinitely — the worker thread was never spawned. Instrumentation revealed start_quicktake_job was returning early before reaching thread.start() because the idempotency guard if qt.status in (PENDING, RUNNING): return qt, False matched the row just created with defaults={'status': STATUS_PENDING, ...}.

Fix: wrap the idempotency check in if not created: so it only applies to pre-existing rows (datapipeline/leai_analysis.py start_quicktake_job). First-run rows fall through to the thread spawn block. Verified: logs now show the full lifecycle (pending → running → ready) and bullets render in the UI.
